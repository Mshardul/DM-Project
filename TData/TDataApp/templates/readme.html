Temporalize: User can temporalize the same table again and again. Unless there still exist one query for the same table, which is not yet addressed by the admin.
Temporalize: Irrespective of what constraints existed for the yet-to-be-temporalized column, they will not be taken into account during temporalization.
Temporalize: We will not be asking the user for the timeline, but will be taking the current time as both - the end time of previously entered row and the starting time of recently entered row.
Admin: When user is trying to temporalize an already temporalized column, Admin will get the notification. But there will be no sql statement for that, as it is already temporalized.


Temporalize Flow
1. User selects DB
2. User selects one of the relation of that db
3. User selects one or more attributes of that relation
4. User click on the 'Temporalize' button
5. An SQL query is generated, and sent to the admin
6. Only admin gets to decide, whether the query is right or not. And hence accepts or rejects it.
6(a). Accept - all the column gets one table each (with 4 columns - including start and end date). The data is obtained from the base table.
6(b). Reject - Query gets deleted from the database.


Insert Flow
1. User selects DB.
2. User selects one of the relation of that db.
3. User inputs values for each of the attributes of the selected relation.
4. User click on the 'Insert' button.
5. A row is inserted in the table.
6. If there exists any temporalized column in the table, new rows are inserted on all of them; plus an update query is also executed (if required).
